# AgeCraft infinite crafting and rarity system

## Core principle
Any owned item can be combined with any other owned item, but outputs are generated by a bounded ruleset rather than a hand-authored recipe table.
That gives effectively unbounded item space without requiring infinite manual content.

## Item model
Each item has:
- `itemId`: deterministic hash-based identifier
- `baseFamilies`: material/technology origins such as stone, fiber, metal, combustion, signal
- `essenceVector`: 5 numeric axes
  - force
  - precision
  - energy
  - intelligence
  - stability
- `traits`: rolled semantic modifiers such as sharp, conductive, fertile, recursive, radiant
- `eraAffinity`: which era problems the item naturally helps solve
- `qualityTier`: broken / crude / workable / excellent / legendary
- `rarityScore`: derived score used for value and discovery logging

## Craft input rules
MVP craft input:
- 2 mandatory input slots
- 1 optional catalyst slot
- later eras can unlock extra slots

Any craft resolves through:
1. normalize input vectors
2. blend vectors with weight based on slot role
3. compute stability penalty from incompatible families
4. apply catalyst mutation if present
5. roll emergent traits using deterministic seed + small random variance
6. output new item if stability > destruction threshold, else salvage

## Why the space is effectively infinite
The system becomes open-ended because:
- vectors are continuous within bounded ranges
- trait pools expand by era and family interaction
- catalysts mutate both vector balance and trait selection
- output IDs include input lineage plus craft seed

So the game does not need a finite recipe book to generate novelty.

## Value and rarity logic
High-value items appear when multiple conditions align:
- output solves an active era bottleneck
- stability is high enough to be usable
- trait combination is rare in the current economy
- lineage contains scarce catalysts or late-era families

This makes powerful items hard to hit by brute force.

## Failure economics
Failed crafts should not feel empty.
Outputs can be:
- salvage fragments
- knowledge points
- partial unstable items usable as catalysts
- market data about what combinations trend toward value

## Anti-chaos guardrails
To keep the system understandable:
- every item shows 3 visible headline properties only
  - primary function
  - stability
  - rarity
- advanced vector details can stay behind inspect mode
- craft previews show probability bands, not exact guarantees
- some traits are mutually exclusive to prevent meaningless combinatorial noise

## MVP implementation rule
For the first build:
- keep family count small: 8-10 families
- keep visible trait pool to 30-40 traits
- keep era bottleneck tags explicit
- store full lineage so future NFT minting can prove provenance
